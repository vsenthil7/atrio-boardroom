# ATRIO Submission Media

Generated pitch deck (.pptx + .pdf) plus **two independent verifications**
that prove the produced PDF says the right things on the right pages.

Mirrors the `atrio/demovideo/` pipeline (creation + verification-a + verification-b)
so the pattern is consistent across all submission artefacts. Re-usable in
future projects: copy this folder, swap the RUBRIC, and you have PDF verification.

## Folder layout

```
submission_media/
  README.md                                 # this file
  atrio-pitch-deck-{stamp}.pptx             # source (49 KB, generated by atrio/scripts/build_pitch_deck.py)
  atrio-pitch-deck-{stamp}.pdf              # final deliverable (~400 KB, generated by atrio/scripts/pptx_to_pdf.ps1)
  _backup/                                  # backups, gitignored
  previews/                                 # PNGs of each slide, gitignored
  verification-a/                           # structural review (24 hard assertions, ~0.5s)
    structural-review.py
    run-verify-a.ps1
    README.md
    reports/structural-review-{stamp}.txt
  verification-b/                           # vision review (OCR + rubric, ~10s)
    extract-pages.py
    grade-pages.py
    grade_pages_lib.py
    run-verify-b.ps1
    requirements.txt
    README.md
    reports/vision-review-{stamp}.json
```

## End-to-end pipeline

```powershell
# 1) BUILD the deck (run from atrio/ root)
python atrio/scripts/build_pitch_deck.py
#    -> submission_media/atrio-pitch-deck-{stamp}.pptx

# 2) CONVERT to PDF (uses MS PowerPoint COM automation on Windows)
pwsh atrio/scripts/pptx_to_pdf.ps1
#    -> submission_media/atrio-pitch-deck-{stamp}.pdf

# 3) VERIFY-A — structural review (~0.5s, 24 hard assertions)
pwsh submission_media/verification-a/run-verify-a.ps1
#    -> submission_media/verification-a/reports/structural-review-{stamp}.txt

# 4) VERIFY-B — vision review (~10s, OCR + rubric)
pwsh submission_media/verification-b/run-verify-b.ps1
#    -> submission_media/verification-b/reports/vision-review-{stamp}.json
```

Both verifications passing on the same commit is the by-construction proof
that the deck is correct. We commit the verification reports alongside the PDF.

## Why this folder exists (not just trust the deck)

Same logic as `demovideo/`: an AI-generated artefact needs an automated
counter-signature before we put it in front of judges. The deck is built
by `build_pitch_deck.py` from project docs; a single content drift in a doc
could silently change a slide. verification-a catches "page missing" or
"wrong page count" failures. verification-b catches "right page, wrong text"
failures. Together they bound the risk.

## Re-using in future projects

```powershell
# In a new project:
cp -r 0010_AT_Hack0021_ATRIO_MilanAIWeek/submission_media/verification-a /path/to/new-project/submission_media/
cp -r 0010_AT_Hack0021_ATRIO_MilanAIWeek/submission_media/verification-b /path/to/new-project/submission_media/

# Then edit two files:
#   verification-a/structural-review.py   -> swap EXPECTED_SLIDES list
#   verification-b/grade_pages_lib.py     -> swap RUBRIC list
```

Everything else (run-verify-*.ps1, extract-pages.py, OCR helpers) is project-agnostic.
